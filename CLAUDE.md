## What this is

A demo gallery for the `vim-web` npm package ŌĆö a 3D viewer for `.vim` (AEC/BIM) models. Each "demo" is a small, self-contained React component showing one feature of the viewer API. The repo's purpose is illustrative: pages are meant to be read as usage examples (every page links to its own source on GitHub), so favor clear, minimal code over abstraction.

## Commands

```bash
npm install
npm run dev          # Vite dev server (defaults to http://localhost:5173)
npm run build        # tsc --noEmit  +  vite build  ŌåÆ dist/
npm run typecheck    # tsc --noEmit only
npm run deploy       # build + publish dist/ to gh-pages branch
```

There is no test suite and no lint script (eslint dev-deps are present but unconfigured). Treat `npm run typecheck` / `npm run build` as the correctness gate.

Note: `replace.js` at the repo root is orphaned ŌĆö no npm script references it, and it edits a `<title>VIM Viewer N</title>` pattern that doesn't exist in `index.html`.

## Two rendering backends

Everything is duplicated across two viewer backends exposed by `vim-web`:

- **Webgl** (`VIM.React.Webgl`) ŌĆö client-side three.js rendering. Create with `createViewer(div)`, then `viewer.load({ url })` or `viewer.load({ buffer })`.
- **Ultra** (`VIM.React.Ultra`) ŌĆö server-backed rendering. **Must call `await viewer.core.connect()` before `viewer.load(...)`** ŌĆö this is the key difference from Webgl.

Backend-specific helper hooks live in `src/webgl/webglUtils.tsx` and `src/ultra/ultraUtils.tsx` (`useWebglViewer`, `useWebglModel`, `useUltra`, `useUltraModel`, etc.). They handle viewer lifecycle (create on mount, dispose on unmount) and expose the live viewer on `globalThis.viewer` / `globalThis.ultra` for poking at the API from the browser console.

## Page / routing architecture

There is no router library. `src/app.tsx` is a hand-rolled router:

- Pages are `Page` objects (`{ name, github, content }`, see `src/page.ts`). `content` is a thunk returning JSX.
- Four registries, each an exported `pages` array in a `pageIndex.tsx`: `src/{webgl,ultra}/{demo,dev}/pageIndex.tsx`. The sidebar groups them as Webgl / Ultra.
- The URL's last path segment is the page slug (derived from the source filename). `app.tsx` keeps the URL in sync via `history.replaceState`, preserving query params.
- Special routes/params: visiting `/dev` swaps the sidebar to the **dev** page sets (internal/error-case demos) instead of **demo**; `?nosidebar=true` hides the sidebar (used for iframe embedding); the Home page reads `?vim=<url>` to override which model loads.

**To add a demo:** create `src/<backend>/<demo|dev>/myDemo.tsx` exporting a component, then register it in that folder's `pageIndex.tsx` with `name`, `github: \`${root}/myDemo.tsx\``, and `content: () => <MyDemo />`. The `github` link must match the real filename ŌĆö `app.tsx` derives the slug from it.

## Demo component conventions

Follow the existing pattern when writing a demo (see `src/webgl/demo/home.tsx`, `localFile.tsx`):

- Hold the container in a `useRef<HTMLDivElement>`; create the viewer in `useEffect` and `viewer.dispose()` in the cleanup. Style the container `className="vc-inset-0 vc-absolute"` (the `vc-` classes come from `vim-web/style.css`, imported once in `main.tsx`).
- After a successful load, call `viewer.framing.frameScene.call()` to fit the camera.
- Sample model URLs (residence, medical tower, the "not a vim" error case, an access-token-protected URL) are centralized in `src/urls.ts`. `urls.ts` also defines `gitRoot`, the base for every page's `github` source link.
- `src/utils/vimCache.ts` provides `fetchVimBuffer(url, onProgress)` ŌĆö an in-memory buffer cache with download-progress callbacks, used by the `useWebglModel`-style hooks.

## Build & deployment specifics

- `vite.config.js` sets `base: '/vim-web-demo/'` ŌĆö must match the GitHub Pages repo name. A custom Vite plugin copies `dist/index.html` ŌåÆ `dist/404.html` so GitHub Pages serves the SPA on unknown routes.
- Build is intentionally **unminified with sourcemaps** (it's a reference app meant to be read).
- `index.html` is dev-server-only (it loads `src/main.tsx`); production HTML is generated by the build.
- Deployment runs via `.github/workflows/deploy.yml` (manual `workflow_dispatch`) or `npm run deploy`.
